## Bug found and fixed mid-run

**Symptom:** Initial first navigation marked `unipixel_url_fired:meta:Lead:/contact*` in sessionStorage, but no `tr` or `admin-ajax` request fired.

**Root cause:** No consent cookie was set. The dispatch function `clientFirstEventTriggered_Meta` short-circuits before firing if `unipixelCheckConsentForEvent()` returns false — but the fire-once helper had ALREADY marked the event as fired. So the flag got set without anything actually firing. If the user later granted consent in the same session, the event would never fire (flag blocks it).

**Fix:** `unipixel-common.js` — `unipixelShouldFireUrlEvent` now checks consent first; if denied, returns false WITHOUT marking. Consent later granted → flag isn't there → event fires correctly.

**Verification:** After consent cookie was manually set and sessionStorage cleared, navigation to `/contact/` produced full firing as expected (Scenario 1).

## Plumbing notes for the agent

- **Hard reload needed** after JS source changes: `unipixel-common.js` is cached by browser (versioned URL via `UNIPIXEL_VERSION = 2.6.5` cache buster). After editing the file, do `Ctrl+Shift+R` or bump `UNIPIXEL_VERSION` for a clean test.
- **Network tracker quirk:** the MCP `read_network_requests` only captures from when first called per tab session. Clearing with `clear: true` resets the buffer. To capture a navigation, call once to start tracking, navigate, wait, then read.
- **HTTPS only:** `updev.local.site` does not respond on `http://`. Always use `https://`.
- **DB credentials** (for state setup via PDO): host=localhost, db=updev, user=root, password=empty. Table prefix `wx4gk_`.
