## Critical incident: my snapshot/restore broke product pages

**Symptom**: `/product/product-1/` returned WordPress critical-error page. Site homepage and shop archive were fine. Product templates and `/wp-admin/admin.php?page=unipixel*` pages also crashed for some classes of request. Discovered ~15 minutes after Round 1 "completed".

**Root cause** (two compounding bugs):

1. **Test artefact (mine)**: At Round 1 setup I wrote a snapshot of `unipixel_logging_options` to `C:/tmp/updev-test-logging-orig.json`. **`C:/tmp/` does not exist on Windows by default.** `file_put_contents()` returned `false` silently — the snapshot was never written. On restore, `file_get_contents()` returned `false`, `json_decode(false, true)` returned `null`, and `update_option('unipixel_logging_options', null)` was called. WordPress stored that as an empty string. Subsequent `get_option('unipixel_logging_options', $defaults)` returned `''` (because the option *exists* with an empty value, the default fallback never fires — that's how WP behaves).

2. **Real plugin bug** (`unipixel-enqueue.php:47`):
   ```php
   $logging_options = get_option('unipixel_logging_options', $logging_defaults);
   $consent_options = get_option('unipixel_consent_settings', $consent_defaults);
   $unipixel_settings = array_merge($logging_defaults, $logging_options, $consent_defaults, $consent_options);
   ```
   No defensive check that `$logging_options` / `$consent_options` are arrays. If either DB value is a string (or any non-array) for any reason, `array_merge` throws `TypeError` → fatal → critical-error page. Stack trace fired on every front-end page render that hits `wp_enqueue_scripts`.

   **Suggested fix**: wrap each `get_option` result with `is_array(...) ? ... : $defaults`. Cheap defensive coding, prevents fatal-on-corrupt-data.

**Resolution**: Wrote a clean default-array `unipixel_logging_options` back via `update_option(...)`. Site recovered immediately. Verified product page loads, `fbq`/`gtag` defined, `UniPixelSettings` is an object.

## More testing-mechanics findings (Round 2)

Adding to the session's accumulated wisdom for future runs:

- **Snapshot files must use a Windows-valid path**. `C:/tmp/` does not exist. Use `C:/Users/RohanKleem/AppData/Local/Temp/` or create a project-local `.claude/testing/_snapshots/` (gitignored) and write there. Adding a snapshot file's existence check post-write would have caught the failure at the right time. Suggested template:
  ```php
  $snapfile = '<absolute-windows-path>.json';
  $written = file_put_contents($snapfile, json_encode($orig));
  if ($written === false) { throw new RuntimeException("Snapshot write failed: $snapfile"); }
  ```
- **Restore must validate the round-trip**. After `update_option(restored)`, immediately `get_option()` and `is_array()` check before declaring the test green.
- **Prefer "set known defaults" over "snapshot-and-restore" for options whose schema is small and stable**. For `unipixel_logging_options` (5 booleans, all default false), it's safer to just write the canonical defaults than to round-trip via filesystem.
- **Browser MCP click-on-submit-button doesn't dispatch native form submit**. For WC Add to cart and similar form-driven flows, locate the button via `javascript_tool` and call `.click()` directly, OR call `form.submit()`. Document as a known pitfall.
- **`location.href` returns `[BLOCKED]` when query string contains anything that looks like cookie/session data**. Use `location.pathname + '?' + new URLSearchParams(location.search).toString()` or just `location.pathname` if you don't need the query.
- **Logging out as admin requires clicking the link via `javascript_tool` (not `computer.left_click(ref)`)** because the admin-bar logout link is hover-revealed and the `find` ref doesn't trigger the parent menu's hover state. Logout URL itself is `[BLOCKED]` from JS read because of the nonce. Pattern: `document.querySelector('a[href*="action=logout"]')?.click()` works.
- **Re-login is a manual step for the user**. Per safety rules, the agent does not enter passwords. End-of-session: leave at `wp-login.php` and tell the user.
- **`facebook.com/privacy_sandbox/pixel/register/trigger/`** is a parallel endpoint Meta is dual-writing. Add to `testing.md` § Verification surfaces as a known beacon, especially when capturing baselines for Meta payloads.

## Fixes applied this session

### 1. Defensive guard around `array_merge` of plugin options
- **File**: `public_html/wp-content/plugins/unipixel/functions/unipixel-enqueue.php`
- **Lines**: 45-50 (added two `is_array()` guards)
- **Rationale**: `get_option(name, $default)` only falls back to `$default` when the option doesn't exist. If the option exists with a corrupt value (string, null, anything non-array), the original code passed that directly into `array_merge`, throwing TypeError → fatal on every front-end render.
- **Verified**: Set option to empty string, reloaded `/product/product-1/`, page rendered cleanly with `UniPixelSettings` object containing 12 keys (defaults merged correctly).

### 2. TikTok brand-casing in platform header H1
- **File**: `public_html/wp-content/plugins/unipixel/admin/admin-wpmenu.php`
- **Lines**: 147-155 (replaced `ucfirst($platform_key)` with a label map)
- **Rationale**: `ucfirst('tiktok')` produces `Tiktok` (single capital). TikTok's brand uses two capitals. Other platforms (Meta, Google, Pinterest, Microsoft) ucfirst correctly, so they were unaffected, but a label map is the right shape for this kind of branding lookup and absorbs any future platform with non-trivial casing (e.g. "X (formerly Twitter)").
- **Verified**: PHP lint clean. Visual verification on the admin page deferred (browser session is logged out; user to confirm post-login).

Both files PHP-linted clean. No other plugin files touched.
